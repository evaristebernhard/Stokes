# Stokes next hard gaps

Date: 2026-05-25

Scope: current `Stokes/Global` and `Stokes/BoundaryChart` Lean files after the
recent endpoint/source-shrink facade waves.  This report is intentionally about
the next hard mathematical gaps, not another generic roadmap.

## Executive conclusion

The current development has a working algebraic Stokes spine:

- `Stokes.Global.Theorem`: finite-sum global Stokes skeleton.
- `Stokes.Global.M8CompactSupportStatement`: compact-support M8 theorem.
- `Stokes.Global.NaturalCompactSupportStokesStatement`: natural compact-support
  wrapper.
- `Stokes.Global.CanonicalNaturalStokes`: canonical names
  `manifoldExtDerivIntegral = boundaryFormIntegral`.

What remains is no longer mainly finite-sum algebra.  The next real gaps are:

1. Construct the endpoint margin fact
   `compactBox_subset_endpointInterior` from the same chart-box construction
   that builds the selected compact boxes and localized outer boxes.
2. Replace the boundary "future target contains selected image box" callback by
   a controlled local-openness/IFT target-selection theorem.  The current fully
   arbitrary quantifier is too strong to follow from local openness alone.
3. After those two are solved, instantiate the global packages from genuine
   compact-support partitions, oriented boundary atlas data, and manifold form
   integration definitions.

The file-sprawl problem is real.  Most new work should extend the existing
constructor/facade families, not add another one-file wrapper.  The only new
Lean files worth adding now are files with genuine geometric lemmas.

## Hard gap 1: endpoint `compactBox_subset_endpointInterior`

The exact current field is in
`Stokes/Global/NaturalCompactSupportEndpointMarginConstructorsAuto.lean`:

```lean
EndpointCompactActiveLocalizedOuterBoxData.compactBox_subset_endpointInterior :
  ∀ x, x ∈ S.selection.compactActiveBoxData.finiteActive.active →
    Set.Icc (S.selection.compactActiveBoxData.lower x)
      (S.selection.compactActiveBoxData.upper x) ⊆
    boxInteriorSupportBox
      (S.endpointMeasureLocalization.localizedInterior.piece x).lowerCorner
      (S.endpointMeasureLocalization.localizedInterior.piece x).upperCorner
```

This is exactly the statement that the selected compact closed coordinate box is
strictly inside the localized outer coordinate box used by the endpoint
localized interior piece.

Existing pieces already cover most of the proof:

- `Stokes.Global.StrictInnerOuterBox`
  - `StrictInnerOuterBoxSelection`
  - `ActiveStrictInnerOuterBoxSelections.ofCompact`
  - `innerIcc_subset_outerInterior`
- `Stokes.Global.CompactActiveBoxes`
  - `CompactActiveBoxData`
  - `CompactActiveBoxData.lower`, `CompactActiveBoxData.upper`
  - `ActiveChartCompactSupportData.toCompactActiveBoxData`
- `Stokes.Global.CompactSupportStrictBufferFromActive`
  - `CompactActiveBoxStrictBufferAlignment`
  - `localized_tsupport_subset_innerIcc`
  - `toLocalizedInteriorFormInnerBoxBuffer`
- `Stokes.Global.EndpointLocalizedOuterBoxFromCompactSelectionAuto`
  - `EndpointCompactActiveLocalizedOuterBoxData.ofCompactActiveBoxStrictBufferAlignment`

So the desired endpoint field is already one projection away from
`CompactActiveBoxStrictBufferAlignment`.  The missing part is not another
projection theorem.  It is a coherent constructor that makes the compact active
box and the endpoint localized outer box come from the same strict inner/outer
selection.

The current construction still allows these choices to drift apart:

- `ActiveChartCompactSupportData.box` chooses a `CompactCoordinateBoxSelection`
  from compactness.
- `ActiveStrictInnerOuterBoxSelections.ofCompact` separately chooses inner and
  outer boxes from compactness.
- `M8MeasureLocalizationData.localizedInterior.piece x` stores lower/upper
  corners separately again.

To close this gap cleanly, build a constructor with one source of truth:

1. Start with the compact coordinate support `coordSupport x` from
   `ActiveChartCompactSupportData`.
2. Choose `StrictInnerOuterBoxSelection (coordSupport x)` once.
3. Build the `CompactCoordinateBoxSelection` for `CompactActiveBoxData.box x`
   using `CompactCoordinateBoxSelection.of_subset` with
   `innerLower`, `innerUpper`, and `K_subset_innerIcc`.
4. Build the localized endpoint piece lower/upper corners from the same
   `outerLower`, `outerUpper`.
5. Prove `compactBox_subset_endpointInterior` by the stored
   `innerIcc_subset_outerInterior`, with only `rfl` or simp-level corner
   equalities.

Recommended Lean location:

- Prefer extending `Stokes.Global.EndpointLocalizedOuterBoxFromCompactSelectionAuto`
  or `Stokes.Global.CompactSupportStrictBufferFromActive`.
- Do not create a new one-declaration endpoint wrapper.

Useful target declarations:

- `CompactCoordinateBoxSelection.ofStrictInnerOuter`
  or a theorem in `StrictInnerOuterBoxSelection` producing
  `CompactCoordinateBoxSelection`.
- `CompactActiveBoxData.ofActiveStrictInnerOuter`
  where `box x` is definitionally the strict inner box.
- `EndpointCompactActiveLocalizedOuterBoxData.ofStrictInnerOuterConstructor`
  consuming the aligned compact active data and localized outer pieces.

Expected proof difficulty: moderate Lean bookkeeping, low mathematical risk.
The main risk is definitional equality of selected boxes.  If definitional
equality is painful, use the already available
`EndpointCompactActiveOuterBoxData` route, which accepts separately named outer
boxes and equality fields to the endpoint pieces.

## Hard gap 2: boundary later-target shrink inequalities

The current facade names the remaining boundary COV geometry in
`Stokes/BoundaryChart/SourceShrinkSelectedCOVFacade.lean`:

```lean
boundaryChartSelectedTargetImageBoxContainedInFutureTargets I x0 x1
  a b targetLower targetUpper y
```

This expands to:

```lean
∀ c d, c 0 = 0 → c ≤ d →
  y ∈ lowerZeroFaceDomain c d →
  boundaryChartLocalInverseData I x0 x1 a b c d →
    Icc selectedTargetImageBox ⊆ Icc (boundaryFaceLowerCorner c)
      (boundaryFaceUpperCorner d)
```

`Stokes.BoundaryChart.SelectedImageBoxContainmentFromShrinkAuto` already gives
the better internal forms:

- `BoundaryChartSelectedTargetContainsLaterTargets`
- `BoundaryChartSelectedTargetContainsLaterTargets.ofTangentBounds`
- `BoundaryChartSourceShrinkInverseTargetBoxData.toSelectedImageBoxContainmentOfAmbientTangentBounds`
- `BoundaryChartSourceShrinkOpenPartialHomeomorphData.toSelectedImageBoxContainmentOfAmbientTangentBounds`

The tangent-bound form says what must be proved:

```lean
∀ laterLower laterUpper,
  laterLower i.succ ≤ selectedLower i.succ
  ∧ selectedUpper i.succ ≤ laterUpper i.succ
```

for all later target boxes under consideration.

Important point: this cannot be derived from local openness or IFT for arbitrary
later target boxes.  If `c d` is any smaller local-inverse target box containing
`y`, it may be strictly inside the already selected target image box.  Then the
desired inclusion is false.  So the right next theorem is not "prove the current
facade predicate from local openness" in full generality.  The right theorem is
"choose later target boxes by a policy that is constrained to contain the
already selected compact image box."

Existing local-openness/IFT tools:

- `Stokes.BoundaryChart.TargetBoxFromIFT`
  - `exists_boundaryChartTargetBoxSelection_of_localOpenness_compactImage`
  - `BoundaryChartLocalOpennessCompactImageCover`
  - `BoundaryChartIFTCompactImageCoverData`
- `Stokes.BoundaryChart.TargetBoxSourceShrink`
  - `nonempty_boundaryChartSourceShrinkMapsToData_of_continuousAt`
  - `BoundaryChartSourceShrinkMapsToData`
- `Stokes.BoundaryChart.TargetBoxSourceShrinkInverse`
  - `exists_lowerZeroFaceDomain_mem_nhds_subset_of_mem_nhds`
  - `BoundaryChartContinuousLocalInverseData.exists_targetShrinkLocalInverseData`
  - `BoundaryChartSourceShrinkInverseTargetBoxData`
- `Stokes.BoundaryChart.TargetBoxSourceShrinkIFT`
  - `BoundaryChartSourceShrinkOpenPartialHomeomorphData`
  - `source_mem_nhds`, `target_mem_nhds`, `mapsTo_target`,
    `inverse_mapsTo_source`

The clean boundary target-selection theorem should look like this:

1. IFT/local homeomorphism supplies an open target neighborhood `U ∈ 𝓝 y` and a
   local inverse on `U`.
2. Source-shrink continuity chooses a smaller source box whose image lies in a
   fixed target neighborhood.
3. Compactness/continuity of the shrunken source image gives a compact image
   box.
4. The selected target/later target box is chosen to contain that compact image
   box and still lie inside the local inverse target neighborhood.
5. Therefore the later-target tangent inequalities follow by construction,
   often by `le_rfl` if the later target is exactly the chosen target box.

Recommended Lean move:

- Introduce a constrained selection lemma, not another callback facade:

```lean
exists_lowerZeroFaceDomain_mem_nhds_subset_and_contains_Icc
```

or a more project-specific boundary version.  It should take:

- `U ∈ 𝓝 y`,
- a compact/tangential image box with strict room inside `U`,
- and return lower-zero corners `c d` such that
  `selectedImageBox ⊆ lowerZeroFaceDomain c d` and
  `lowerZeroFaceDomain c d ⊆ U`.

If the selected image box does not have strict room around `y`, first select a
strictly larger ambient box around it.  That is the honest geometric statement.

Recommended Lean location:

- Put the general box-neighborhood lemma in
  `Stokes.BoundaryChart.TargetBoxSourceShrinkInverse`.
- Put the source-shrink-to-COV constructor in
  `Stokes.BoundaryChart.SelectedImageBoxContainmentFromShrinkAuto` or
  `Stokes.BoundaryChart.SourceShrinkSelectedCOVFacade`.

Avoid trying to prove the existing `boundaryChartSelectedTargetImageBoxContainedInFutureTargets`
from only:

- `y ∈ lowerZeroFaceDomain c d`,
- `boundaryChartLocalInverseData ... c d`.

That statement is too strong without a restriction on how `c d` is chosen.

Expected proof difficulty: medium-high.  The local topology is standard, but
the API must avoid an impossible universal statement.

## Suggested next 6-agent wave

1. Endpoint strict-box constructor:
   build `CompactCoordinateBoxSelection` / `CompactActiveBoxData` directly from
   `StrictInnerOuterBoxSelection`, so compact active boxes are the strict inner
   boxes by construction.

2. Endpoint localized outer alignment:
   extend `EndpointLocalizedOuterBoxFromCompactSelectionAuto` to produce
   `EndpointCompactActiveLocalizedOuterBoxData` from the strict inner/outer
   constructor plus localized piece corner equalities.

3. Boundary constrained target-box lemma:
   prove a neighborhood box-selection lemma that returns target boxes containing
   a fixed selected image box while staying inside the local inverse target
   neighborhood.

4. Boundary source-shrink IFT constructor:
   construct `BoundaryChartSourceShrinkOpenPartialHomeomorphData` from the
   actual local homeomorphism/IFT data, with `mapsTo_target`,
   `inverse_mapsTo_source`, and the target-containment inequalities built into
   the selected boxes.

5. Orientation bridge audit/proof:
   try to instantiate `BoundaryChartMathlibOrientationManifoldData` from the
   currently available mathlib orientation/manifold API; if impossible, produce
   the exact missing upstream theorem statement.

6. Global constructor integration:
   work from `NaturalCompactSupportStokesInput` backward and list/prove the
   first constructor that creates it from compact support, selected partition,
   target-image source-shrink COV data, measure localization, and artificial
   cancellation without exposing raw endpoint fields.

## File-count guidance

Do not create more micro-files for:

- field projections,
- `rfl` simp lemmas,
- one-step constructor wrappers,
- import-only adapters.

Preferred edit targets:

- Endpoint geometry:
  `EndpointLocalizedOuterBoxFromCompactSelectionAuto`,
  `CompactSupportStrictBufferFromActive`,
  `StrictInnerOuterBox`.
- Boundary target geometry:
  `TargetBoxSourceShrinkInverse`,
  `SelectedImageBoxContainmentFromShrinkAuto`,
  `SourceShrinkSelectedCOVFacade`.
- Public imports:
  use `BoundaryChartCOVFacade`; add a global compact-support endpoint facade
  only if it actually replaces several direct imports.

The next progress metric should not be "number of new declarations".  It should
be the number of theorem-facing assumptions removed from
`NaturalCompactSupportStokesInput` and
`NaturalCompactSupportEndpointSelectedReconstructionBaseSources`.
